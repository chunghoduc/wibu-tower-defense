# Passive Skill Tree — Massive PoE-Style Expansion

**Date:** 2026-06-20
**Status:** Design approved (autonomous session)

## Problem

The passive tree currently has **78 hand-authored nodes** across 8 regions radiating
from `grid-start`. Players earn **1 skill point per hero level**, capped at level 100
(the only point source — `hero.ts:94`; forget/reset only refund). So a max player has
~100 points for 78 nodes and can allocate **the entire tree plus more**. There is no
opportunity cost, no build commitment, and the tree is trivially "solved."

The user wants a Path-of-Exile-style tree where:

1. **A max-level player can complete only ONE path** (build commitment / opportunity cost).
2. **A max-level player can never allocate even 1/5 of the tree** (the tree dwarfs the
   point budget — PoE allocates ~120 of ~1,800 nodes ≈ 7%).

## Goals

- Grow the tree to **≥ 880 nodes** so 100 points ≈ 11% (< 20% — requirement #2 with margin).
- Make **fully traversing one region lobe cost ~90 points** so one path = your whole
  build (requirement #1).
- **Zero save migration / zero lost progress**: every existing node ID, position, and
  stat stays exactly as authored. New nodes only *extend* outward.
- Keep the tree **navigable** at 10× scale: pan + zoom camera.
- Respect repo law: every source file < 500 code lines; texture keys only via
  `assetKeys.ts` (N/A here — no art); 0 runtime cycles; deterministic (no `Math.random`).

## Non-Goals

- **No new art / no ASSET_VERSION bump.** Nodes are vector-drawn (`passiveGridGlyphs.ts`).
- **No change to point economy** (still 1/level, cap 100). The expansion alone creates
  scarcity; we do not touch XP, level cap, or point grants.
- **No change to the stat pipeline** (`stats.ts heroStatPipeline`), choice resolution
  (`passiveChoice.ts`), jewel sockets, forget/reset, or save schema fields. Generated
  nodes are ordinary `PassiveNodeDef`s consumed by the existing seams unchanged.
- **No rebalance of existing node values.** (A future pass may tune; out of scope here.)
- **No hand-authoring 800 nodes.** They are generated by a pure deterministic builder.

## Approach

### 1. Preserve + Extend (not replace)

`PASSIVE_NODES` becomes `buildExtendedPassiveTree(BASE_NODES)` where `BASE_NODES` is the
current 78-node hand-authored array (unchanged). The builder appends generated nodes that
attach to existing **anchor** nodes (each region's outermost keystone/notable). Because no
existing ID, position, neighbor, or stat changes, **all current saves keep working with no
migration** — `unlockedNodes`, `nodeChoices`, socketed jewels all still resolve.

### 2. Region lobes (one angular sector each)

The 8 regions (`brawler, arcane, warden, predator, tactician, phantom, conduit, prestige`)
each own a ~45° angular **sector** around `grid-start` (12,9). Generated nodes for a region
are laid out on a **polar grid** within that sector — `ring` (distance from center, ~3..18)
× `angle` (within the sector) → `gridX/gridY` — expanding the existing inner authored nodes
outward into a large lobe. Sectors don't overlap, so lobes never collide, and the result
reads as a PoE-style wheel.

Each lobe grows to **~95–105 generated nodes** (8 × ~100 ≈ 800) on top of the 78 authored
= **~880 total**.

### 3. Lobe internal structure (the commitment lever)

From a region's anchor keystone, the generator grows:

- A **main spine**: a chain of small `path` "travel" nodes (small `increased` stat from the
  region's theme pool) — these are the toll you pay to reach power.
- **Sub-branches** off the spine (3–4 per lobe), each ending in a **notable** (a bigger
  stat bag), and the longest ones in a **new outer `keystone`** (`more` multiplier +
  `effectId`) — the payoff.
- **Jewel sockets** (2–3 per lobe) and **mastery choice nodes** (1–2 per lobe, 3 options
  each) sprinkled at branch joints.
- **Deeper = stronger:** outer-ring nodes carry slightly larger numbers than inner ones, so
  the long walk is worth it — but the spine's many travel nodes mean reaching the outer
  keystones of *one* lobe costs ~90 points, leaving nothing for a second lobe.

Cross-lobe travel remains only through the existing `prestige-gate-25/50/75/90`
(`unlockAtLevel`-gated) bridges far from center — you can't cheaply hop sideways.

### 4. Pure deterministic generator

`src/data/passiveTreeGen.ts` (+ `passiveTreeSpec.ts` for the compact per-region specs):

- `buildExtendedPassiveTree(base: PassiveNodeDef[]): PassiveNodeDef[]` — returns base ⧺
  generated, validated via `validatePassiveNode`.
- Generation is **fully deterministic**: a tiny seeded RNG (reuse `core/rng.ts Rng`) keyed
  per region, **no `Math.random`/`Date.now`** (repo law — they throw in tests anyway).
- Generated node IDs are namespaced `gen-<region>-<ring>-<i>` so they can never collide with
  authored IDs.
- The builder enforces invariants (asserted by tests): all neighbors resolve; the whole
  graph is connected to `grid-start`; no two nodes share a `gridX,gridY`; node count ≥ 880;
  no generated ID collides with an authored ID; every lobe is reachable.

### 5. Navigable UI (pan + zoom)

`PassiveGridScene` currently maps grid→fixed pixels in a fixed rect with no camera control —
unusable at 60-wide grids. Changes:

- A **tree camera** renders the Graphics tree; supports **drag-to-pan** and
  **wheel/pinch-to-zoom** (clamped zoom 0.4–1.6; pan clamped to tree bounds + margin).
- A **second fixed UI camera** renders the side panel + buttons + title so they don't scroll.
  Tree objects are ignored by the UI camera and vice-versa (`camera.ignore`).
- On open, the tree camera **centers on the player's allocation frontier** (or `grid-start`
  if nothing allocated) so a returning player lands on their build.
- Hit-testing converts pointer screen→world via `camera.getWorldPoint` before the
  nearest-node search (existing `handleGridClick` math, fed world coords).
- Pure pan/zoom clamping math extracted to `src/scenes/passiveTreeCamera.ts` (testable);
  the side panel extracted to `src/scenes/passiveGridPanel.ts` so `PassiveGridScene` stays
  < 500 code lines after the additions.

## Testing (TDD)

Pure-logic tests drive the build (RED→GREEN):

1. **`passiveTreeGen.test.ts`** — node count ≥ 880; all neighbors resolve; graph connected
   from `grid-start`; no duplicate `gridX,gridY`; generated IDs disjoint from authored IDs;
   determinism (two builds deep-equal); every region lobe present and reachable.
2. **Commitment invariant** — pure BFS cost: the cheapest set of nodes to reach a region's
   outer keystone is ≥ ~85 points; the union cost to reach **two** different regions' outer
   keystones is > 100 (can't afford two paths at max).
3. **Coverage invariant** — `100 / PASSIVE_NODES.length < 0.20` (the headline requirement,
   locked as a test).
4. **`passiveTreeCamera.test.ts`** — zoom clamps to [0.4,1.6]; pan clamps within bounds;
   center-on-frontier returns the centroid of allocated nodes (or grid-start centroid).
5. **Regression** — existing `passiveChoice`, `heroStatsChoice`, `saveManager`,
   `getReachableNodes`, `canForgetNode` tests stay green (no behavior change for authored
   nodes). A new test asserts every BASE node still exists with identical stats after extend.

## Risks

- **UI performance** drawing ~880 nodes + edges each redraw. Mitigation: the tree is drawn
  to a single `Graphics` once per allocation change (not per frame); pan/zoom is camera
  scroll/zoom (no redraw). Verify FPS in the CDP playtest.
- **Layout collisions** at scale. Mitigation: polar-per-sector placement + a duplicate-cell
  assertion in tests fails the build if any two nodes share a cell.
- **`PassiveGridScene` > 500 lines** after camera wiring. Mitigation: extract panel + camera
  modules (planned above) before adding.
