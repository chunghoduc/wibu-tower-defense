# Skill: pixel-art-cleanup

Every sprite generated by the Z-Image-Turbo image API comes out as a 1024×1024
smooth raster image. Before it can be used as pixel art it must go through this
cleanup pipeline to reach the NES/SNES aesthetic the game requires.

## Why this is mandatory

AI image models apply sub-pixel anti-aliasing. At 1024×1024 this is invisible, but
when the image is downscaled to 32×32 or 192×192 those blended edge pixels smear
into muddy grey fringe. Cleanup must happen **before** downscaling, not after.

## Pipeline steps (in order)

### Step 1 — Background removal (white → transparent)

```bash
node src/cli.ts generate <key>    # handles this automatically
```

The CLI uses `sharp` to flood-fill the white background to alpha.
Threshold: any pixel with R > 240, G > 240, B > 240 → alpha 0.

If the background is not pure white (gradient, shadow, noise), the image model
prompt needs fixing first — add "flat white background, no shadow, no gradient".

### Step 2 — Anti-aliasing removal via PixelRefiner

Open **PixelRefiner** at `https://pixel-refiner.app` (runs entirely in-browser, no upload):

1. Drag the raw 1024×1024 PNG in.
2. Set **Pixel Grid** → `Auto` (detects grid from frequency analysis).
3. Set **Edge Mode** → `Snap to Grid` (replaces blended edges with nearest solid colour).
4. Set **Palette** → `NES` or `Custom` — use the custom option to import the project's
   64-colour palette from `src/core/palette.ts` as a hex list.
5. Export → PNG → save as `output/<id>-clean.png` in the designer workspace.

**When PixelRefiner is not available** (offline, running headlessly), the CLI's own
palette quantization step in `node src/cli.ts generate <key>` applies K-means
colour reduction to 64 colours. This is less precise but always available.

### Step 3 — Nearest-neighbour downscale

The CLI handles this:

```bash
node src/cli.ts generate <key>
```

Uses `sharp.resize(targetW, targetH, { kernel: 'nearest' })` — never bicubic,
never lanczos. Any interpolation reintroduces anti-aliasing.

Target sizes per kind:

| Kind | Single frame | Full sheet (7 frames) |
|------|-------------|----------------------|
| tower | 192×192 | 1344×192 |
| hero | 192×192 | 1344×192 |
| boss | 256×256 | 1792×256 |
| enemy | see manifest | N/A (1 frame) |
| item | 96×96 | N/A (1 frame) |
| vfx | 128×128 | N/A (1 frame) |

### Step 4 — Palette quantization (64 colours, NES discipline)

```bash
node src/cli.ts generate <key>    # posterize step included
```

The canonical 64-colour palette lives in `src/core/palette.ts`. After downscale,
each pixel is snapped to its nearest palette entry (Euclidean distance in sRGB).

**Colour discipline rules (from the game's art spec):**
- Every tower's palette must include its rarity accent colour:
  - Common: muted grey-steel
  - Magic: cobalt blue
  - Rare: royal purple
  - Legendary: molten orange-gold
  - Unique: crimson red
- Enemies use a "corrupted" palette: desaturated, cooler, darker than tower colours.
- VFX: bright saturated only; transparent where not the effect itself.

### Step 5 — Frame consistency check (from Sprite-Pipeline conventions)

For animated sprites (tower/hero/boss), all 7 frames must share the **same palette
and the same character silhouette origin point**. Drift between frames reads as
visual glitching in Phaser's animation system.

Before accepting the frames, run this visual check:
```bash
node src/cli.ts gallery --id <key>   # renders an HTML preview of all frames side-by-side
```

Open the HTML file and visually inspect:
- [ ] Silhouette stays within the same bounding box across all frames
- [ ] No colour appears in one frame but not in another (unless intentional VFX)
- [ ] The "hurt" frame (last) shows a clear recoil or damage state

If frames are inconsistent, re-generate the bad frame(s) using the same `seed` base
(add the frame index as offset: seed + frameIndex) and the same prompt + action suffix.

## What good output looks like

- Crisp 1-pixel outlines with no grey fringe
- Clearly readable silhouette at the target size (test by squinting)
- Rarity accent colour visibly present
- Transparent background (no white halo)
- `verify` reports: ✓ dimensions ✓ frames ✓ alpha

## Common problems and fixes

| Problem | Fix |
|---------|-----|
| Grey fringe around character | Re-run PixelRefiner with Edge Mode → Snap to Grid |
| White halo after bg removal | Lower threshold in CLI (`--bg-threshold 220`); or re-generate with "flat white background" in prompt |
| Too few colours, looks blobby | Increase image API `steps` to 11; add detail cues to prompt |
| Colour drift across frames | Re-generate drifted frames with same seed; quantize after all frames are generated as a batch |
| Palette accent colour missing | Add explicit accent colour to prompt: "with a purple glow on the weapon" for Rare rarity |
